# Claude Code との開発プロセス — TenShot

このドキュメントは「**どうやって作ったか**」を残すもの。
TenShot の開発を通じて見えた、**Claude Code（AI コーディング支援）と人間の協働設計**の知見。

これは UX 判断（[ux-design.md](./ux-design.md)）でも技術判断（[architecture.md](./architecture.md)）でもなく、**開発プロセスそのものに対する設計**の話。

---

## 学んだ最重要ポイント

> **人間がボトルネックにならない設計にすると、開発速度が一気に上がる**

これが TenShot 開発で得た一番大きな学び。
**技術選定でも UX でもなく、Claude Code との協働の組み立て方**。

---

## 最初のフロー（うまくいかなかったやり方）

開発初期はこういう流れだった：

1. Claude Code に「メニューバーアプリを作って」と要件を伝える
2. Claude Code がコード（Swift ファイル）を生成する
3. **自分が Xcode を開いて、生成されたコードを取り込み、Xcode の GUI 設定（Signing & Capabilities、entitlements、Info.plist）を手作業で変える**
4. ビルドしてみる、エラー出る
5. エラーを Claude Code に貼り付ける、「これを直して」
6. Claude Code が「Signing & Capabilities タブを開いて〜」と指示してくる
7. **また自分が Xcode で操作する**
8. 1〜7 を延々と繰り返す

### この方式の問題

**自分が Claude Code の手になっている**。

Claude Code は「ここを直してください」と言える。
でも実際に直すのは人間。
人間が Xcode を開いて、設定タブを探して、チェックボックスを ON にして、Save する……

このループは、思った以上にコストが高い。
Claude Code の応答速度がいくら速くても、人間が物理的に操作する時間がボトルネックになる。

特に Xcode の GUI 設定は **項目が深い階層に埋もれている** ことが多く、設定を探すだけで数分かかる。
これが何十回と繰り返されると、開発時間の大半が「設定を探す→チェックを入れる→保存する」に消える。

## ブレークスルー：Claude Code に Xcode の設定ファイルを直接触らせる

ある日気づいた：**Xcode のプロジェクト設定は、実体はテキストファイルである**。

```
TenShot.xcodeproj/project.pbxproj   ← ビルド設定・ファイル参照のすべて
TenShot/TenShot.entitlements        ← 権限設定
TenShot/Info.plist                  ← アプリ情報
```

Xcode の GUI で操作している項目は、**裏でこれらのテキストファイルを書き換えている** だけ。
GUI は単なるラッパーで、実体はテキスト編集。

ということは、**Claude Code に「ここを直してください」と言わせる必要がない**。
**Claude Code が自分で `project.pbxproj` を直接書き換えればいい**。

人間がやることは、Xcode を再度開いて変更を確認するだけ。

## 改善後のフロー

1. Claude Code に要件を伝える
2. Claude Code が Swift コードを生成 + `project.pbxproj` / `entitlements` まで全部書き換える
3. **自分は Xcode でビルドボタンを押すだけ**
4. エラーが出たら Claude Code に貼り付ける、Claude Code が `project.pbxproj` まで触って解決する
5. 再度ビルド

人間の介在ポイントが「**ビルドボタンを押す**」と「**エラーを貼り付ける**」だけになった。
**Xcode の GUI 操作がほぼゼロ**。

体感として、**1 機能の追加にかかる時間が 1/3〜1/5 になった**。
特に「権限まわり」「entitlements」「ビルド設定」など、コード以外の周辺作業が一気に楽になる。

## 一般化：「人間経由」をどこから減らせるか

この経験から得られる原則：

### 原則1：人間が AI のアドバイスを実行するパターンは、人間がボトルネックになる

```
[AI: アドバイスを生成（数秒）]
    ↓
[人間: アドバイスを読む（10秒）]
    ↓
[人間: 該当の場所を探す（1〜数分）]
    ↓
[人間: 操作する（数十秒）]
    ↓
[人間: 結果を AI に報告する（数十秒）]
    ↓
[AI: 次のアドバイスを生成（数秒）]
```

人間の時間が AI の時間の **10〜100 倍** かかる。

### 原則2：AI が直接触れるインターフェース（テキストファイル、CLI）に問題を翻訳できれば、人間の介在を減らせる

| 人間経由 | AI 直接 |
|---|---|
| GUI 設定（Xcode の Signing タブ等） | 設定ファイル直接編集（`project.pbxproj`） |
| ボタン操作（メニュー → 何々を選択） | スクリプト化（`xcrun`、`xcodebuild` コマンド） |
| 報告と承認のループ | AI が自分で完結する範囲を広げる（実行＆検証まで AI 側でやる） |

### 原則3：応用可能な場面

これは Xcode に限らない。あらゆる「**GUI が前提のツール**」に応用できる発想：

- **PowerShell / AppleScript** で GUI アプリを自動操作
- **JSON / YAML 設定ファイル** で GUI 設定を代替
- **dotfiles** でツールの初期設定を自動化
- **IaC（Infrastructure as Code）** でクラウド設定を GUI から脱却

**AI に直接触らせられる「テキスト面」を見つける** ことが、開発速度の鍵になる。

## 余談 1：Chrome 拡張試作からの方針転換

最初の試作は **Chrome 拡張機能** だった。スクショの大半が Chrome 内（Web の参考画像）だったから、自然な選択肢に見えた。

しかし作りながら気づいた：**Chrome の外側（VSCode、Figma、Slack、ターミナル）でもスクショは撮るし、見比べる**。
Chrome 拡張機能では物理的にこれらに干渉できない。

そこで方針を変えた。「Chrome 内外を問わず使う」なら、OS レベルに張り付くしかない。**macOS ネイティブアプリ** へ移行。

これは「**作りながら問題の輪郭が見えてくる**」典型例。
要件定義の段階で「こういうのが欲しい」と思っていたものは、実装してみると「実はもう一つ大きい問題を解いていた」と気づくことがある。
**最初の試作を捨てる勇気が、結果的に正しいアプリを生む**。

Claude Code を使った開発では、この「捨てる」コストが極端に低い。
ゼロから書き直してもらっても数時間で同じ機能まで戻せる。
**捨てる判断のハードルを、AI が下げてくれている**。

## 余談 2：要件定義は「こんなのがあったらいいな」で始められる

開発開始時、自分が用意した要件はこんなレベル：

```
- メニューバーに常駐してほしい
- スクショを最大10枚くらいストックできればいい
- クリックひとつでコピーできれば最高
- 暗めのデザインで、画面に馴染んでほしい
- 透過できれば他のウィンドウの邪魔にならない
- シンプルでいい。機能は少なくていい
```

これをそのまま Claude Code に投げた。
「要件定義書を書いてください」のような形式は不要。**ざっくりした言葉で十分伝わる**。

形式の整った要件定義は、AI 時代には逆にオーバーヘッドになる場合がある。
**「ぼんやりしたまま投げて、AI と対話して輪郭を出す」** 方が、初期の試行錯誤コストが低い。

## 余談 3：個人開発者一人で App Store 申請まで完走できる時代

このプロセスを通じて得た実感：

- **アイデアを思いついた日に**、Xcode プロジェクトの雛形ができる
- **翌日には**、主要機能が動く
- **数日で**、ビルド・テスト・申請までいける
- 個人開発者一人で、コードを書かずに、App Store にアプリを出せる

「コードが書けないから諦める」という障壁が、Claude Code によって完全に消えた。

要るのは：
- 「**こういうのが欲しい**」という言葉
- 「**人間経由を減らす**」という設計感覚
- 「**最初の試作を捨てる**」勇気

この 3 つだけ。

技術的な深い知識は、必要になった瞬間に AI に教えてもらえばいい。
**「全部自分で理解してから始める」モードを脱却すること** が、AI 時代の個人開発のスタート地点になる。

## まとめ：3 つのレイヤーの設計

TenShot の開発を振り返ると、設計を 3 つのレイヤーで考えていたことが見える。

### レイヤー1：プロダクト設計（[ux-design.md](./ux-design.md)）

「**ユーザーにどう使ってほしいか**」を決める層。
TenShot で言えば、「邪魔しない」「溜めない」「クリップボードを壊さない」の 3 原則。

### レイヤー2：技術設計（[architecture.md](./architecture.md)）

「**プロダクト設計を実現するために何をどう組むか**」を決める層。
TenShot で言えば、AppKit / NSPanel / ScreenCaptureKit の選定、Application Support での永続化など。

### レイヤー3：開発プロセス設計（このドキュメント）

「**プロダクトと技術を生み出す『作り方』そのもの**」を決める層。
TenShot で言えば、Claude Code に Xcode の設定ファイルまで触らせる協働パターン。

**多くの個人開発者はレイヤー1と2しか考えない。** でも AI 時代にはレイヤー3 の設計が、開発速度と完走率を決める。
**「AI とどう組むか」を最初から設計に含める** こと。これが 2026 年以降の個人開発のスタンダードになる。

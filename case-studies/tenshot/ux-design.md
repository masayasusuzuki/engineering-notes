# UX 設計思想 — TenShot

このドキュメントは「**なぜこの形にしたのか**」を残すもの。
機能仕様ではなく、判断の背景を書く。

---

## なぜ作ったか — 「消しちゃいけない画像」を消した日

Mac で作業していると、スクリーンショットがデスクトップに溜まり続ける。
ファイル名は `スクリーンショット 2026-04-12 14.32.18.png` のような自動生成。中身を判別する手がかりは、ファイル名にも、Finder のサムネイルにも、ほとんどない。

ある日、デスクトップが散らかりすぎて、20枚くらい溜まっていたスクショをまとめて全削除した。
その中に、撮り直しできない重要な画像が混ざっていた。

「ファイル名で判別できないから、全消ししか選択肢がない」
「全消しすると重要なものまで巻き込まれる」

この瞬間が TenShot の出発点。
**「溜まってから困る前に、溜まらない仕組みを作る」** という方針はここから来ている。

---

## なぜクリップボード履歴を欲しがったか

もう一つの根っこは Mac の標準クリップボードの仕様。
**1枚しか保持できない**。画像 A をコピー → 画像 B をコピーすると、A は失われる。

3〜5枚の画像を並べて見比べたい場面（資料作成、デザインレビュー、コーディング中の参考画像）では、これが致命的に不便だった。

Windows には `Win + V` でクリップボード履歴が標準搭載されている。Mac にはない。
**「Mac でも複数の画像を一時的に並べて持っておく場所が欲しい」** という欲求が固まった。

「デスクトップが汚れる問題」と「クリップボードが1枚しか持てない問題」、両方に効く解として「**最大10枚を一時的にストックするパレット**」という形が見えた。

---

## なぜ既存ツールを使わなかったか

正直に書くと、CleanShot X / Paste / Pastebot / PixelSnap などの既存ツールは **一切試していない**。

代わりに **AI と壁打ちしながら自分の要件を言語化することから始めた**。
「こういうのが欲しい」というぼんやりした感覚を、AI に投げて返ってきた質問に答えながら、徐々に具体化していった。

結果として「既存ツールに自分を合わせる」のではなく、「自分の使い方を起点に作る」アプローチになった。
これは効率の良いやり方ではない（既存ツールの調査ステップを飛ばしている）。
でも、自分の使い方にぴったりハマる形のものを最初から作れる。トレードオフ。

---

## なぜ Chrome 拡張機能ではなく macOS ネイティブアプリなのか

第一試作は **Chrome 拡張機能** だった。
スクショの大半を Chrome 内で扱っていた（Web の参考画像、デザインリファレンス）から、自然な選択肢に見えた。

しかし作りながら気づいた：**Chrome の外側でもスクショは撮るし、見比べる**。
VSCode のエラー画面、Figma のデザイン、Slack のスクショ、ターミナル出力、Notion のメモ……
Chrome 拡張機能では物理的にこれらに干渉できない。

そこで方針を変えた。Chrome 内外を問わず使うなら、**OS レベルに張り付くしかない**。macOS ネイティブアプリへ移行。

この判断が後から効いてくる：

- メニューバー常駐 → どこから呼んでも 0 秒で出せる
- ドラッグ＆ドロップで他アプリと直接連携できる
- 範囲選択スクショで画面のどこからでも切り取れる
- フルスクリーン中の他アプリの上にも乗せられる

Chrome 拡張のままだったら、これらは全て不可能だった。
**「最初の試作を捨てる」判断**が、結果的に正しいアプリを生んだ。

## なぜ「10枚」なのか

8 でも 12 でもなく 10。理由は2つ。

### 直近10枚あれば実用上十分

「昨日撮ったスクショ」「3日前のスクショ」を呼び戻したいことは、実際にはほとんどない。
使うのは「**今さっき撮ったやつ**」「**今日のうちで何回か行き来しているやつ**」。
10枚あれば、現実的なワークフローはほぼカバーできる。

### 「無制限」は新しいデスクトップになる

「上限なし」にすると、結局元の問題（溜まり続ける、判別不能、消すのが怖い）が再発する。
**意図的に上限を設けることが、解になる**。

`maxImages = 10` は実装上の数字だが、思想としては **「これ以上は要らない」** という線引き。

## なぜ FIFO（最古を自動削除）なのか

11枚目を撮った瞬間に、最古の1枚が自動的に消える。

これは **「ユーザーに削除判断を強いない」設計**。

選択肢は他にもあった：

- **LRU**（最近使ってないものから消す）：「最近使った」を判定するロジックが必要、挙動が予測しにくい
- **手動削除のみ**：また「重要なものを巻き込んで消す」事故が起きる

FIFO は **予測可能で、認知負荷が最小**。
「新しいのが入る = 古いのが出る」というシンプルな約束。
ユーザーは古い画像が消えることを心配せずに次のスクショを撮れる。

「重要な画像を巻き込んで消した」原体験への、設計レベルでのアンサー。

## なぜ右端の縦長パレットなのか

画面右端に幅 300px の縦長パレットを固定する。

### 横型ではバーと当たる

Mac の画面は、上にメニューバー、下に Dock、画面の縦中央付近に各アプリのウィンドウ、というレイアウト。
**横型のパレットを置く場所がない**。横一列に並べると Dock やメニューバーと干渉する。

### 右側が一番見やすい

多くのアプリは UI が左寄り（VSCode のサイドバー、Finder のサイドバー、ブラウザのブックマーク）。
**右端は最も「邪魔されにくい」場所**。
左側にパレットを置くと、ほぼ全てのアプリのメイン UI を覆ってしまう。

### 縦型なら 10 枚を並べて一覧できる

横型で10枚並べるには画面を横切る必要があり、見るたびに視線が大きく動く。
縦型なら一瞥で全体を把握できる。スクロールも自然。

```swift
// PanelWindowController.swift
let x = screen.visibleFrame.maxX - w  // 画面右端に固定
let y = screen.visibleFrame.minY      // 画面の縦いっぱい
```

## なぜメニューバー常駐なのか

普通のアプリのように Dock に入るのではなく、メニューバーに `10Shot` というラベルだけを置く。

理由：**「立ち上げ」というアクションを最小化するため**。

スクショ管理の道具は「使うときだけ呼び出す」もの。
アプリスイッチャーで切り替える、Dock からクリックする、というステップは重すぎる。

メニューバー左クリックでパレット表示、もう一度クリックで非表示。
**起動・終了のコストがほぼゼロ**になる。

加えて、立ち上げた直後の存在感も控えめにしている。
画面右端に静かに出てきて、何もアニメーションせず、フォーカスも奪わない。
「**呼んだから来た**」感を最小に抑える。

```swift
// AppDelegate.swift
statusItem = NSStatusBar.system.statusItem(withLength: NSStatusItem.variableLength)
button.title = "10Shot"
```

## なぜ半透明スライダーがあるのか

縦型でスマートにしてもなお、画面右端 300px を占有することは作業の邪魔になりうる。
特に **VSCode のような IDE** で右側にミニマップやサブパネルがある場合、TenShot のパレットがそれを隠す。

「常時表示するけど存在感は下げられる」状態を作るために、不透明度スライダーを付けた。
背景の VSCode が透けて見える状態にしておけば、必要なときだけパレットを意識し、それ以外は無視できる。

```swift
// PanelViewController.swift
let slider = NSSlider(value: 0.92, minValue: 0.2, maxValue: 1.0, ...)
```

最小値を 0.2（20%）に設定している。完全透明にはしない。
完全に透明にすると操作不能になる（クリックできない）し、存在を忘れて二重起動の事故も起きうる。
**「うっすら見える」を維持する**のが、共存設計のキモ。

これは「常駐型ツールが既存の作業環境にどう溶け込むか」という、メニューバーアプリ特有の課題への回答。

## なぜドラッグ＆ドロップなのか

`Copy` ボタンでクリップボードへ送る、というオーソドックスな操作だけでなく、
**「画像をパレットから他アプリへ直接ドラッグ」** もできるようにした。

これは前述の「Mac クリップボードが 1 枚しか持てない」問題への直接の答え。

`Copy` ボタンを押すと、クリップボードを上書きしてしまう。
ドラッグなら、**クリップボードの中身を壊さずに**、他アプリへ画像を届けられる。

例：

- クリップボードに「重要なテキスト」が入っている
- TenShot のパレットから画像を Slack の入力欄にドラッグ
- 画像が貼られる、テキストはクリップボードに残ったまま

`NSFilePromiseProvider` を使ってファイルとして渡しているので、受け取り側はファイルとして処理できるアプリ（Finder、Slack、Notion、Mail 等）なら何でも対応する。

## なぜ「Buy Me a Coffee」リンクを削除したか

開発初期、パレット下部に小さく「☕ 開発者にコーヒーを奢る」という Buy Me a Coffee へのリンクを置いていた。
無料アプリなので、気に入ってくれた人がいたら任意で支援してもらう想定。

**App Store 申請時、このリンクが理由で Apple から差し戻しを受けた**。

App Store 規約上、デジタルコンテンツの外部決済への誘導は原則禁止。
Buy Me a Coffee はその範疇に入る（「開発者を支援する」というのも、Apple の解釈では「デジタル取引」になる）。

リンクを削除して再申請したら通った。

教訓：**Mac App Store で配るアプリには、寄付・スポンサー誘導を組み込めない**。
そういう動線は、別経路（Web サイト、SNS、リポジトリ README）から行う必要がある。

このリポの README やこのドキュメントから Web サイト等への誘導は問題ない。
アプリ内に組み込むのが NG というだけ。

## なぜグラスモーフィズム＋ダーク基調なのか

濃紺ベース（`rgb(8, 8, 20)`）に薄白の半透明カードを浮かせるグラスモーフィズム。

macOS のシステム UI（コントロールセンター、通知センター、Spotlight）と同じデザイン言語。
**OS 標準の見た目に揃える**ことで、「あとから入れたサードパーティアプリ」感を消し、最初からそこにあったような佇まいにする狙い。

```swift
// PanelViewController.swift より
private enum Glass {
    static let bg1 = NSColor(red: 8/255, green: 8/255, blue: 20/255, alpha: 1)
    static let bg2 = NSColor(red: 15/255, green: 15/255, blue: 40/255, alpha: 1)
    static let cardBg = NSColor(white: 1, alpha: 0.07)
    static let cardBorder = NSColor(white: 1, alpha: 0.15)
    // ...
}
```

ダーク基調にした理由は、**コーディング中・デザイン作業中はダークテーマで作業していることが多い**から。
背景アプリ（VSCode、Figma）と色味の連続性を保つ。

なお、初期にはコルクボード風（`CorkTextureView`）の試作もあった（写真をマスキングテープで留める見立て）。
これは結局採用しなかった。「**スクショ管理の道具**」というクールな機能性に対して、「コルクボード」というメタファーは情緒的すぎた。
グラスモーフィズムの方が「道具」としての佇まいに合う。

リポにはこの試作の残骸が `CorkTextureView.swift` として残っている。
意図的に削除していないのは、**設計プロセスの記録としての価値**を残すため。

## まとめ：TenShot を貫く設計原則

書き起こしてみると、すべての判断が3つの原則に集約される。

### 1. 「溜めない」を仕組みで作る

10枚上限・FIFO・終了時の永続化…
ユーザーに整理を強いず、**システムが勝手に直近だけを残す**設計。
これは「全消しで重要な画像を消した日」の経験への、構造レベルでのアンサー。

### 2. 「邪魔しない」を最大化する

メニューバー常駐・右端縦長・半透明スライダー・フォーカスを奪わない NSPanel…
**作業の主役は他のアプリ**であり、TenShot は脇役に徹する。
「使うときだけ意識される、それ以外は背景に溶ける」存在を目指している。

### 3. 「クリップボードを壊さない」直接連携

ドラッグ＆ドロップでクリップボードを介さず画像を渡せる。
Mac の標準クリップボードの制約を回避する設計。
**OS の制約を補完する道具**であることを明確にしている。
